# 삼성·하이닉스 → KOSPI200 ETF Lead-Lag 차익거래 알람 봇(토스증권, 한국투자증권)

> ## ⚠️ 고위험 트레이딩 (HIGH-RISK TRADING)
> 이 프로젝트는 **초·분 단위 단기 차익거래**를 다룹니다. 차익거래 창은 매우 짧고,
> **호가 스프레드·수수료·슬리피지·체결 실패·급격한 역행**으로 빠르게 손실로 전환될 수
> 있습니다. 레버리지·인버스 ETF는 일일 변동성이 지수의 2배이며 장기 보유 시 복리
> 손실(decay)이 누적됩니다. **본 모듈은 알람·리서치 전용이고 자동주문을 포함하지
> 않으며, 어떤 수익도 보장하지 않습니다. 투자 권유가 아닙니다.** 실거래는 전적으로
> 본인 책임입니다. 매우 위험하며 원금 전액 손실이 가능한 트레이딩 기법입니다.

> 빅2(삼성전자+SK하이닉스 ≈ KOSPI200의 55%)의 **추세 확인된** 움직임이
> KOSPI200 추종 현물/레버리지/인버스 ETF를 **선행(lead)** 하는 구간을 탐지해
> 콘솔 + 로그파일로 알람을 보내는 봇과, 그 가설을 데이터로 검증하는 모듈.
>
> **신호 2종**: ① lead-lag(빅2 선행 vs ETF 미반영) · ② 괴리율(시장가 vs iNAV
> 프리미엄/디스카운트 되돌림). **시세 소스**: KIS WebSocket(H0STCNT0) 실시간 체결 →
> REST 폴링 → MOCK 자동 폴백.

밀리세컨드 HFT가 아니라 **30초 / 60초 / 2분 / 3분 / 5분** 타임프레임의 추세를 보고,
그중 **여러 타임프레임에서 동시에 지속되는** 추세만 신뢰한다(초단기 노이즈 차단).

주의: 한국투자증권은 API 신청후 3일 동안 초당 3건으로 api 호출을 제한하고 있다.(보안 및 사고 위험성), TOSS 역시 유사 정책이 있을 것으로 보인다. 
늘 이 부분은 초보 개발자가 테스팅을 하다가 증권사 서버를 DDoS를 때리듯 패킷을 날려 죽이는 사고가 난다.

## 1. 가설과 검증 설계

**가설**: `삼성·하이닉스 합성 지수(composite)`의 추세 확인된 움직임이 KOSPI200 추종
ETF를 선행한다. ETF가 아직 못 따라온 구간 = 차익거래 창.

**검증 방법** ([verify_leadlag.py](verify_leadlag.py)):
1. 틱 CSV를 고정 그리드로 리샘플(ffill).
2. 빅2 로그수익률을 지수 가중 합성 → composite(지수 프록시).
3. 각 ETF 수익률을 배수(leverage)로 정규화한 뒤 composite와 **lag별 교차상관**.
4. **lag>0에서 상관 피크 → composite가 ETF를 선행 = 가설 지지.**

**방법론 검증 결과** (셀프테스트가 4초 지연을 의도 주입한 데이터):

```
ETF                      배수     피크상관    피크lag        해석
KODEX 200                 1     1.000       4s     지수 선행(가설지지)
KODEX 레버리지             2     1.000       4s     지수 선행(가설지지)
KODEX 200선물인버스2X      -2     1.000       4s     지수 선행(가설지지)
→ 주입한 4초 lead-lag를 정확히 복원. 검증 파이프라인 유효.
```

> 이는 **방법론이 lead-lag를 잡아낸다는 증명**이다. 실거래 판정은 반드시
> `node bot.js --record`로 모은 **실시장 ticks.csv**에 같은 검증을 돌려서 한다.

## 4. 시그널 발화 조건 (signal.js / DEFAULT_CONFIG)

### ① lead-lag 신호 (`evaluate`)
1. 빅2 추세가 기준 타임프레임(`60s`)에서 **confirmed**(|수익률|≥0.12% & 방향 일관성≥0.6).
2. ETF `gap = leverage×지수수익률 − 실제ETF수익률`의 |값| ≥ **0.15%**.
3. 위 조건이 **`['60s','2m']` 두 타임프레임에서 동시에 같은 방향**으로 성립.
4. 통과 시 방향(UP/롱 후보 · DOWN/숏·회피 후보)과 함께 알람.

### ② 괴리율 신호 (`evaluateDisparity`, NAV 수신 모드에서만)
1. `disp = (시장가 − NAV)/NAV` 의 |값| ≥ **0.3%(band)**.
2. 최근 `lookback=5` 샘플이 같은 부호로 band를 넘어 **지속**(consistency≥0.8) → 초단기 노이즈 차단.
3. 프리미엄(disp>0, 고평가) → 하락 되돌림(DOWN) · 디스카운트(disp<0, 저평가) → 상승 되돌림(UP).

설정값은 모두 `DEFAULT_CONFIG`에서 조정 가능. 실시장에 맞춰 임계값·확인 타임프레임을
보정하라(예: 변동성 큰 날은 gap·band 임계 상향, 확인 타임프레임에 `3m` 추가).
알람 쿨다운은 `(유형·코드·방향)` 단위라 lead-lag와 괴리율이 서로를 막지 않는다.

## 5. 전략의 원리와 원칙

### 5.1 원리 (왜 빅2가 ETF를 선행하는가?)

1. **쏠림 구조**: 삼성전자+SK하이닉스가 KOSPI200의 약 55%. 지수의 절반 이상이 단 두
   종목으로 결정되므로, 빅2의 방향은 곧 지수의 방향에 가깝다.
2. **체결 우선순위**: 외국인·기관의 지수 베팅과 프로그램(차익) 매매는 유동성이 가장 큰
   빅2부터 체결되는 경향이 있다. 즉 **정보가 빅2 호가에 먼저 찍히고**, 지수 산출 →
   ETF LP의 호가 갱신 → iNAV 반영으로 전파되는 사이에 미세한 시차가 생긴다.
3. **두 신호의 메커니즘**
   - **① lead-lag**: 빅2 합성지수(composite)는 *방금 일어난* 지수 움직임의 선행 추정치.
     ETF 시장가가 아직 그만큼 못 움직였으면 `gap`이 벌어진다 → ETF가 따라올 방향을 예고.
   - **② 괴리율(iNAV)**: ETF 시장가가 실시간 공정가치(NAV)에서 벗어난 정도. 지속적
     프리미엄/디스카운트는 LP·차익거래에 의해 **NAV로 되돌아가는** 경향이 있다.
4. **왜 30초~5분인가**: 밀리세컨드 영역은 LP·HFT가 이미 지배한다(개인 API로는 승산 없음).
   반대로 너무 긴 추세는 차익이 아니라 방향성 베팅이 된다. **초·분 단위의, 그리고
   일정 시간 지속되는** 시차만이 개인이 관측·검증 가능한 구간이라는 판단이다.

5. **시스템의 한계**: 일반 투자자의 투자 패턴이 초단타가 될 경우 증권사 서버와 코스콤 사이 레이턴시를 비록하여 FDS에서 거래를 지연시키거나 차단할 가능성이 높다. 초단타 매매는 증권사 서버에 DDoS 공격을하는 것과 유사한 스트레스를 준다. 채결 엔진 자체가 다운되는 경우 민사뿐 아니라 형사 소송의 대상이 될 수 있다.

### 5.2 원칙 (운용 규칙)

1. **지속성 우선**: 단일 바·단일 틱이 아니라 **여러 타임프레임에서 동시에 지속**되는
   추세만 신뢰한다(`['60s','2m']` 동시확인 + 방향 일관성). 초단기 노이즈는 버린다.
2. **비용 차감 후 판단**: 모든 신호는 **수수료+스프레드+슬리피지를 뺀 순(net)** 기준으로
   재평가한다. 순 우위가 없으면 신호가 떠도 거래하지 않는다(백테스트로 사전 확인).
3. **알람까지만, 결정은 사람**: 자동주문 미포함. 봇은 후보를 제시할 뿐 진입·청산·손절은
   사람이 판단한다. 주문 연결 시에도 사전검증(매수가능금액·가격제한·호가)을 강제한다.
4. **저빈도 운용**: 폴링 1초 이상, rate limit 준수. **HFT를 흉내내지 않는다**(아래 5.3 FDS).
5. **방향·손절 규율**: 차익 신호는 *되돌림/추격*에 대한 짧은 베팅이다. 진입 시 보유시간과
   손절선을 미리 정하고, 역행 시 즉시 청산한다. 레버리지·인버스는 비중을 더 줄인다.
6. **실데이터 재검증**: MOCK/합성 결과는 방법론 시연일 뿐. 실매매 전 `--record` 실데이터로
   교차상관·백테스트를 반드시 다시 통과시킨다.

## 6. 한계와 리스크 (반드시 읽을 것)

- **실행 비용**: 실제 lead-lag 창은 호가 스프레드 + 수수료 + 슬리피지보다 작은 경우가 많다.
  검증 단계에서 **비용을 차감한 순(net) gap**으로 다시 평가할 것.
- **인과 아님**: 교차상관은 선행/동행을 보여줄 뿐 수익을 보장하지 않는다. composite는
  지수의 55%만 설명(잔차 45% 미관측)하므로 **예측자이지 정답이 아니다**.
- **레짐 의존**: 빅2 쏠림이 심한 장에서 신호가 강해지고, 분산장에선 약해진다.
- **레버리지·인버스 decay**: 2x·-2x 상품은 일일 복리 구조라 횡보·변동장에서 가치가
  잠식된다. 단기 차익 외 보유는 추적오차·롤오버 비용에 노출된다.
- **체결·역행 리스크**: 신호가 떠도 호가가 비어 체결이 안 되거나, 진입 직후 빅2가 역행하면
  ETF가 따라오기 전에 손실이 먼저 확정될 수 있다.
- **자동주문 미포함**: 본 모듈은 알람까지만. 주문 연결 시 [../../Toss/docs/Toss_OpenAPI_Guide.md](../../Toss/docs/Toss_OpenAPI_Guide.md)
  시나리오 1·5의 사전검증(매수가능금액·가격제한·호가) 단계를 반드시 거칠 것.
- **데이터/약관**: TradingView는 저빈도 연구용으로만. 실매매 시세는 증권사 API로.

### 6.1 증권사 FDS·약관 리스크 (초단타 시도 시 계좌·API 제한 가능)

증권사 서버에는 **FDS(이상거래 탐지 시스템, Fraud Detection System)** 와 API 이용약관·
rate limit이 있다. 다음과 같은 **초단타(스캘핑)·고빈도 패턴은 이상거래로 탐지되어
계좌·Open API 이용이 경고·제한·정지되거나, 발생한 거래가 무효화·불이익 처리될 수 있다.**

- **과도한 호출/주문 빈도**: 허용된 rate limit(예: 계좌·시세 API 초당 호출 한도)을 넘는
  반복 호출, 1초 미만의 초단타 주문 반복.
- **시세 교란성 패턴**: 즉시 취소를 전제한 허수성 주문(스푸핑), 자기 계좌 간 자전거래성
  매매, 호가를 흔드는 반복 정정·취소.
- **비정상 트래픽**: 동일 토큰 다중 세션 남용, 비정상적으로 짧은 간격의 토큰 재발급 루프.

**이 프로젝트의 방어 설계**: ① 폴링/실시간을 **1초 단위**로 묶고 밀리세컨드 주문을
하지 않는다. ② **알람 전용 — 자동주문·자전·허수 주문을 일절 만들지 않는다.** ③ 신호는
**일정 시간 지속된 추세**만 채택해 발화·행동 빈도를 낮춘다. ④ 토큰은 만료 마진을 둔
선제 재발급으로 캐시 공유([../../Toss/docs/Toss_OpenAPI_Guide.md](../../Toss/docs/Toss_OpenAPI_Guide.md) 시나리오 6).

> 실매매로 확장하더라도 **거래소·증권사의 시장질서 교란행위 규정과 API 약관을 우선**한다.
> 단기 차익을 노린 고빈도 매매는 수익 이전에 **계좌 정지·법적 책임**의 리스크가 더 크다는
> 점을 분명히 인지하고, rate limit·주문 빈도·패턴을 약관 한도 내로 유지할 것.
